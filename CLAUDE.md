# Правила работы и оркестрация

Полные правила протокола общения и рабочего процесса — в `~/.claude/rules/protocol.md` и `~/.claude/rules/workflow.md`. Они user-level rules: автозагружаются в каждую сессию целиком — я знаю их с самого начала, сюда не дублируются.

Напоминалки `/remind-protocol` и `/remind-workflow` инжектируют те же файлы повторно (освежить в длинной сессии); копий контента нет.

## Текст для людей

Когда пишу текст, который читает человек — **обязательно вызываю `/remind-text-style`** перед написанием. Два уровня:

- **Маскировка символов** (— → -, нет умных кавычек, нет backticks вокруг идентификаторов и т.п.) — применяется **везде**: коммиты, МР-комменты, ответы коллегам, описания тикетов, документация, тех. комменты в коде. Не палиться нигде.
- **Стиль** (короче, без преамбул, 1 факт + 1 фикс) — применяется **только** к коротким интерактивным текстам наружу: МР-комменты, ответы коллегам. НЕ применяется к коммит body, документации — там нужно подробно объяснять.

Исключение для маскировки: внутри настоящих ``` блоков кода ``` символы значимы, не трогаем.

# Технические приёмы

## Большие файлы

**Никогда не читать файл целиком без проверки размера.** Стратегия: `ls -lah` → `head -30` → `grep + head` → якоря.

Запрещено: `cat huge.log`, `Read(file)` без проверки, `grep` без `| head`.

## Shell

- **zsh**: сложные `$()` в циклах — оборачивать в `bash -c '...'`
- **macOS**: нет `timeout` — использовать `run_in_background=true` или `gtimeout`
- **Долгие команды** (>30с): запускать с `run_in_background`. Мониторить через `tail -5`, не через BashOutput без фильтра.

## Временные файлы

Все временные скрипты, логи `run_in_background`, черновики и diff'ы — в **`<cwd>/.tmp/`** (в текущей рабочей директории).

- `mkdir -p .tmp` перед использованием (идемпотентно)
- **Запрещено**: писать в `/tmp/`, `/var/tmp/`, `~/.claude/tmp/`. В корпоративных средах `/tmp` + `nohup` часто мониторится безопасниками
- `.tmp/` уже в глобальном gitignore (`~/.config/git/ignore`) — в коммит не попадёт

# GitHub Gists

**Публичные гисты — только с явного подтверждения.** По умолчанию всегда secret. Если пользователь просит публичный — переспросить: напомнить что содержимое будет проиндексировано, проверить нет ли корпоративных данных (названия проектов, тикеты, MR, внутренние хосты). Только после явного «да, делай публичным» — создавать с `--public`. Secret гист доступен по прямой ссылке без авторизации — в большинстве случаев этого достаточно.

# Git

**Для коммитов ВСЕГДА используй `/commit`.** Не коммить напрямую.

Исключение — внутренности коммит-флоу: fix-коммиты цикла `/review` (шаг 5 скилла) и `/finalize` (шаг 3c) делаются напрямую через git commit — `/commit` изнутри них не вызывается.

**Подпись коммитов.** Каждый созданный коммит привязывается к сессии: строка `<short-hash> <sessionId> <YYYY-MM-DD> <title>` в `~/.claude/projects/<slug>/memory/commit-sessions.md` (всегда, любой репо). Trailer `Ref: <sessionId>` в message — только если проектный CLAUDE.md явно разрешил (личные репо); дефолт — без следов в репо. Каноника алгоритма — `/commit` секция «Подпись коммита». Вернуться к сессии коммита: grep хэша по журналам → claude --resume <UUID>.

**Дефолт: `git push` требует явного разрешения пользователя.** Если /commit или /review предлагают push, или ветка feature и кажется что push логичен — ВСЁ РАВНО спрашивай. Force-push, обычный push, push в feature-ветку — всё требует явного «да, пушь» от пользователя в текущем сообщении.

**Per-project override.** Конкретный проект может ослабить (или ужесточить) это правило через свой `CLAUDE.md` или `.claude/CLAUDE.md`. Например: личный пет-проект может разрешить auto-push main в origin после clean review; рабочий клиентский репозиторий — наоборот, запретить любые авто-операции, даже merge. Если в проектном файле override не описан — действует дефолт «спрашивай».

# Worktrees

**Любая задача с правкой файлов — в worktree, без исключений.** Только чтение/поиск/прямой ответ — напрямую.

- **Я сама правлю** → `EnterWorktree` (в начале), `ExitWorktree` (в конце)
- **Делегирую агенту** → `Agent(isolation: "worktree")`

> Подробнее: `/remind-workflow` → секция «Изоляция через worktree»

# Tech Debt

**Фиксируй техдолг СРАЗУ через `/techdebt`** — не спрашивай "стоит ли записать?", просто записывай.

Путь хранения определяет сам `/techdebt` (см. `~/.claude/skills/techdebt/SKILL.md` шаг 0 — канонический алгоритм).

При исправлении — удали файл записи и строку из INDEX.md.

# Нити сессии

**Отложенная тема — сразу строка в журнал.** Когда идея или вопрос откладывается («ИДЕЯ», «потом», «обсудим позже», «в очередь»), допиши строку в `~/.claude/projects/<slug>/memory/session-threads.md` (slug по main-репо): `<sessionId> | <YYYY-MM-DD> | <нить одним предложением>` (sessionId — полный UUID текущей сессии, тот же что возвращает session-id.sh). Это формализация правила «не забывать оффтопные идеи»: журнал переживает compaction. Разбор и триаж — `/finalize` шаг 5c.

# Session ID и compaction

Compaction создаёт новый `--resume <UUID>`. Старый UUID из summary становится «родительским», новый — текущим.

**Правило:** перед каждой передачей `sessionId` в sub-agent (review-judge, /commit, /finalize, /techdebt с parentSessionId) — пересчитывай UUID через ancestor процесса, не доверяй памяти и summary. Алгоритм одинаковый везде: `bash ~/.claude/skills/techdebt/session-id.sh`.

`/techdebt` дополнительно дописывает id в worktree-local `<git root>/.tmp/active-tds.md` — это детерминированный источник для `/finalize`, не зависящий от UUID. Не трогай файл руками.
