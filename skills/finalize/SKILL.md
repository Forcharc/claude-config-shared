# Finalize

Закрой все техдолги текущей сессии, разбери flow-feedback и нити сессии.

## Шаг 1: Найти техдолги этой сессии

Собираем кандидатов из двух независимых источников и объединяем — UUID-логика хрупкая (compaction меняет UUID, sub-agent'ы пишут со своим), worktree-локальный файл — детерминированный fallback.

1. **Worktree-local история** (приоритетный источник). Файл `<git root>/.tmp/active-tds.md`, куда `/techdebt` дописывает id каждого созданного техдолга. `<git root>` — `git rev-parse --show-toplevel`. Прочитай — это список `td-NNN`, созданных в текущем worktree (живёт пока жив worktree, не шарится между ветками).

2. **UUID текущей сессии** — извлеки через ancestor процесса (`--resume <UUID>` в commandline parent Claude процесса):
   ```bash
   bash ~/.claude/skills/techdebt/session-id.sh
   ```
   Раньше использовался `ls -t ~/.claude/projects/*/*.jsonl` — это ненадёжно в форках (мог вернуть UUID parent сессии если она писала свежее).

3. Путь к tech-debt директории — см. `/techdebt` шаг 0.

4. Собери `td-*.md` файлы, удовлетворяющие любому из условий:
   - id присутствует в `<git root>/.tmp/active-tds.md`, ИЛИ
   - frontmatter `originSessionId` == текущему UUID.

   Записи без `originSessionId`, не упомянутые в active-tds и с другим UUID — не трогай.

5. **Compaction-edge-case.** Если `active-tds.md` пуст и по UUID ничего не нашлось, проверь маркер начала текущей сессии — если в `~/.claude/projects/*/<currentUUID>.jsonl` видна фраза `continued from a previous conversation`, значит сессия — продолжение после compaction. Извлеки UUID родителя из summary (или найди его как самый свежий .jsonl с mtime до текущей сессии) и повтори фильтр по нему. Спроси пользователя «нашлись td с UUID родителя, обрабатывать?» перед тем как трогать.

Если после всех источников список пуст — доложи пользователю и выйди.

## Шаг 2: Показать и триаж

Выведи найденный список. Для каждого техдолга вместе с user'ом определи категорию:

- **Чинить** — нет блокера, решение применимо сейчас.
- **Блокер** — есть реальная зависимость (другая фича, ожидание архитектурного решения, внешнее условие). Не чиним сейчас, но запись остаётся.
- **Неактуально** — стал неактуален (фича отменена, код переписан, проблема решилась сама). Не чиним и не храним.

Подожди явного подтверждения. По умолчанию все «чинить», если user не уточнил иначе.

## Шаг 3: Обработать по категориям

**Важно:** шаги 3b и 3c правят файлы напрямую. Убедись что работаешь внутри worktree — если ещё нет, `EnterWorktree` перед началом.

### 3a. «Чинить» — делегировать параллельно

Для каждого техдолга — `implementer` (правила делегирования, worktree-изоляция, передача данных через файлы — см. `/remind-workflow`). В промпте: путь к `td-NNN-*.md`, задача «применить Предлагаемое решение, удалить сам .md-файл, НЕ трогать INDEX.md (skill обновит INDEX пакетно в Шаге 5)».

### 3b. «Блокер» — пометить и оставить

Для каждого техдолга в этой категории:
1. Открой `td-NNN-*.md` через Read.
2. Добавь в конец секцию `## Блокер` с явным описанием: от чего зависим, почему сейчас не чиним, при каких условиях можно вернуться.
3. В frontmatter поменяй `status: open` → `status: blocked`.
4. Файл остаётся на месте, в INDEX тоже остаётся (строка не убирается в Шаге 5).

### 3c. «Неактуально» — удалить

Для каждого техдолга в этой категории:
1. Удали файл: `rm <путь>/td-NNN-*.md`.
2. Запомни id и slug для Шага 5 (там убирается из INDEX).
3. Отдельный коммит с сообщением вида `Remove stale td-NNN: <slug>` и body — почему стал неактуален (1-2 строки контекста).

Шаг 3c не требует `implementer`'а — это чисто файловая операция, делаю сама (внутри активного worktree).

## Шаг 4: После каждой закрытой пачки (категория «чинить») — `/review`

Коммити правки и вызывай `/review` на HEAD. **Обязательно передавай `repo=<worktree-path>`** — иначе если cwd не совпадает с worktree, /review проверит HEAD основного дерева и fix-коммиты цикла /review уйдут в неправильный ref. Всё остальное в зоне `/review`.

## Шаг 5: Merge и обновление INDEX

Когда worktree'ы clean — мердж в основное дерево. Одним коммитом убери из INDEX.md строки для:
- закрытых техдолгов из категории «чинить»,
- удалённых техдолгов из категории «неактуально».

Строки из категории «блокер» НЕ убираются — они остаются в INDEX с пометкой `status: blocked`. Push регулируется проектным CLAUDE.md (см. шаг 8 в `/commit`): если в `<repo>/.claude/CLAUDE.md` явно разрешён auto-push — пушь, иначе НЕ пушь.

Дополнительно очисти `<git root>/.tmp/active-tds.md`: удали из него строки id, попавшие в «чинить» (закрыты) и «неактуально» (удалены). Строки «блокер» оставь — техдолг живёт, должен попасть в следующий `/finalize`. Если файл стал пустым — можно удалить.

## Шаг 5b: Разобрать flow-feedback

Прочитай `~/.claude/projects/$(printf '%s' "$HOME/.claude" | sed 's|[/.]|-|g')/memory/flow-feedback.md` (журнал проблем флоу, который пишут агенты). Если непустой — триаж каждой строки вместе с user'ом: починить промт сейчас (мелочь — прямо здесь же), записать td, или отбросить как шум. Разобранные строки удали из журнала; неразобранные (отложенные user'ом) оставь.

## Шаг 5c: Нити сессии

Собери нерешённые нити текущей сессии из двух источников:

1. **Журнал**: `~/.claude/projects/<slug>/memory/session-threads.md` (slug по main-репо текущего проекта) — строки текущей сессии, фильтр по полному sessionId (UUID) в начале строки. Если по текущему UUID строк нет или мало — проверь UUID родителя (compaction-edge-case: сессия может быть продолжением после compaction, строки могли писаться с родительским UUID; алгоритм поиска родителя — тот же что в шаге 1 пункт 5).
2. **Ретроспектива**: перечитай диалог сессии из своего контекста — отложенные идеи («ИДЕЯ», «потом обсудим»), незакрытые вопросы, данные user'у обещания. Compaction мог вырезать середину диалога — поэтому журнал первичен, ретроспектива дополняет.

Объединяя источники: если нить из ретроспективы совпадает по смыслу со строкой журнала — считай её одной нитью, не дублируй.

Объединённый список покажи user'у и триажь каждую нить вместе с ним: **сделать сейчас** (только мелочь — крупное оформляй отдельной задачей после финализации) / **в память** (backlog-файл проекта) / **в td** (`/techdebt`) / **отбросить**. Дефолт при молчании user'а — «в память».

После триажа: для каждой разобранной нити сначала выполни действие (запись в память / td), потом удали строку из журнала. Чужие строки не трогай.

## Подписи

Все коммиты, создаваемые этим скиллом (3c, шаг 5), подписываются по канонике `/commit` секция «Подпись коммита» (журнал всегда; trailer — только при opt-in проектного CLAUDE.md).

## Шаг 6: Отчёт

Короткий итог пользователю: закрыто X (чинили), помечено блокерами Y, удалено как неактуальные Z, не закрылись W и почему.
