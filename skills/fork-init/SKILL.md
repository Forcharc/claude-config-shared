# Fork Init

Вызывается user'ом сразу после форкания чата. Объясняет мне (Claude) что я форк, и помогает не помешать основной (parent) сессии.

## Контекст

**Форк** — полная копия чата на момент форкания. Я (форк) унаследовала всю историю и контекст основной сессии. Параллельно **в основном чате** продолжает работать другая инстанция Claude — над теми же или другими задачами.

Главные риски форка:
- Я могу подумать что «продолжаю работу из истории» и начать править файлы которыми занимается основная сессия.
- Worktree из истории чата (например `keen-borg-f51fa9`) принадлежит **основной сессии** — мои Edit/Write/commit там создадут гонку.
- Глобальные `~/.claude/*` (правила, agents, skills, project memory) — общий ресурс. Параллельные правки могут конфликтовать.

## Шаг 1: Запомнить scope форка

User даёт новый scope при форке (примеры: «работай над промтами», «исследуй UI», «пиши документацию»).

**Правило:** scope форка ограничен этой задачей. Не возвращаться к задачам из унаследованной истории, даже если они «не доделаны».

Если user не дал scope явно — спросить «какая моя задача в этом форке?» прежде чем что-то делать.

## Шаг 2: Создать новый worktree

Если задача форка предполагает правки кода в проекте — обязательно `EnterWorktree` для нового изолированного дерева. Имя — связанное с задачей форка, **не унаследованное** из истории чата.

Если задача форка только чтение/исследование/правка `~/.claude/*` — worktree не обязателен.

**ЗАПРЕЩЕНО:** Edit/Write/commit в worktree из истории чата. Чтение оттуда — можно.

## Шаг 3: Правила работы в форке

Применяй пока находишься в форк-сессии:

1. **Worktree основной сессии не трогать.** Только Read/Grep — для понимания контекста.
2. **Глобальные `~/.claude/*` (правила, agents, skills) править только если scope форка явно про это.** Иначе риск гонки с основной сессией.
3. **Project memory `~/.claude/projects/<slug>/memory/`** — правки только если scope форка явно про memory. Прежде чем писать — `ls -t` посмотреть нет ли свежих правок от основной сессии.
4. **`/finalize` и `/techdebt` уже знают про форки** — определяют свой UUID через `ps`-ancestor (см. `--resume <UUID>` в commandline parent Claude процесса). Это работает корректно: техдолги форка имеют свой UUID, parent'а — свой.
5. **`git push`** — те же правила что в обычной сессии: только с явного разрешения user в текущем сообщении.
6. **При сомнениях** «трогать ли это?» — спросить user.

## Шаг 4: Подтвердить активацию

Кратко доложить user:
- Какой scope я записала.
- Создан ли новый worktree (если да — название).
- Что готова работать с учётом форк-правил.

## Аргументы

`$ARGUMENTS` — текстовое описание scope от user. Если пусто — спросить.
