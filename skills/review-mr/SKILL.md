# REVIEW-MR — ревью GitLab merge request'а

Полный флоу ревью МР от сбора контекста до публикации черновиков комментариев. Пользователь САМ публикует комменты — ты только готовишь черновики через `/txt`.

> **TODO**: сейчас скилл покрывает только GitLab (через `glab`). Для GitHub нужен отдельный инструмент через `gh`. Если ссылка на github.com — предложи пользователю создать такой скилл или сделать pr review руками.

## Общие правила

- **Только блокеры и важное.** Никаких nit'ов, стилевых мелочей, «хорошо бы», если пользователь явно не попросил.
- **Пользователь сам публикует.** Ты только готовишь черновики. Не пытайся публиковать комменты через API без явного запроса.
- **Каждый коммент обсуждается отдельно.** По одному, ждать одобрения. По умолчанию выдавай прямо в чат — вызывай `/txt` только если нужен отдельный файл (длинный коммент, много спецсимволов).
- **Соблюдай `/remind-protocol`**: русский, сквозная нумерация вопросов, варианты с ⭐ Фаворитом при развилках, ждать решения пользователя.
- **Никогда не используй AskUserQuestion** — вопросы обычным текстом.

## Шаг 0. Распарсить URL МР

Из ссылки извлеки:
- **Хост** (например, `gitlab.example.com`, `gitlab.com`)
- **Путь репо** (например, `group/subgroup/project`)
- **`<id>` МР** (число после `/merge_requests/`)

Эти значения понадобятся во всех следующих шагах. Далее в инструкции `<id>` и `<path>` — это они.

## Шаг 1. Preflight — авторизация к GitLab

Посмотри на домен в URL МР.

### Если корпоративный GitLab (несколько инстансов с разными токенами)

У пользователя может быть два GitLab инстанса с разными токенами, активным из них тот, что раскомментирован в `~/.zshrc`. Токены переключаются вручную.

Проверь:

```bash
grep -E "GITLAB_TOKEN|GITLAB_URL" ~/.zshrc
glab auth status 2>&1
```

Если активный токен относится к другому инстансу (или 401 Unauthorized) — есть два выхода:

1. **Оверрайдить env в каждой команде** (быстро, без правки конфига):
   ```bash
   GITLAB_TOKEN=<corp-token> GITLAB_URL=https://gitlab.example.com/ glab mr view <id> --repo <path>
   ```
2. **Переключить токены в `~/.zshrc`** (закомментировать один блок, раскомментировать другой) — только с явного разрешения пользователя, потому что затрагивает его шелл-конфиг.

Фаворит: вариант 1, если ревью разовое. Если пользователь регулярно работает с этим инстансом — предложи переключить.

### Если другой GitLab инстанс

Стандартный флоу:

```bash
glab auth status
```

Если не залогинен — предложи пользователю:
```bash
glab auth login --hostname <host>
```

## Шаг 2. Получение данных о МР

> **Важно:** если на шаге 1 выяснилось, что активный `GITLAB_TOKEN` относится к другому инстансу, префиксуй **каждую** `glab`-команду ниже env-оверрайдом: `GITLAB_TOKEN=<correct> GITLAB_URL=<correct> glab ...`. Без этого все команды вернут 401.

```bash
# Инфа о МР
glab mr view <id> --repo <path>

# Диф в файл (не инлайн — он может быть большой)
glab mr diff <id> --repo <path> > .tmp/mr-<id>.diff

# Метаданные для fetch ветки (url-encoded path: слеши → %2F)
glab api projects/<url-encoded-path>/merge_requests/<id>
```

Последняя команда возвращает JSON. Извлеки из него `source_branch`, `target_branch`, `sha`, `diff_refs.base_sha` — поле `base_sha` вложенное в `diff_refs`. Пример:

```bash
glab api projects/<url-encoded-path>/merge_requests/<id> | python3 -c "
import json, sys
d = json.load(sys.stdin)
print('source:', d['source_branch'])
print('target:', d['target_branch'])
print('sha:', d['sha'])
print('base:', d['diff_refs']['base_sha'])
"
```

Проверь размер дифа до чтения:

```bash
wc -l .tmp/mr-<id>.diff
grep -cE "^(\+\+\+|---)" .tmp/mr-<id>.diff   # количество файлов
grep -E "^(\+\+\+|---) " .tmp/mr-<id>.diff   # список файлов
```

## Шаг 3. Запрос контекста тикетов

В описании МР часто есть ссылки на тикеты (JIRA, YouTrack, Linear, и т.п.). Автор полагается на контекст из тикета — без него ревью будет поверхностным.

**Попроси пользователя скопипастить содержимое тикетов.** Ты в них сама не попадёшь.

Формулировка:
> В описании МР ссылки на тикеты \<список\>. Скинь содержимое каждого (описание, acceptance criteria, важные комменты) — иначе не пойму мотивацию изменений.

Если тикетов в описании нет или автор их не приложил — продолжай без них, но отметь это ограничение в объяснении МР.

## Шаг 4. Изоляция — worktree и локальная ветка

```
EnterWorktree name: mr-<id>-review
```

Затем в новом worktree:

```bash
git fetch origin <source_branch>:mr-<id>
git checkout mr-<id>
git log --oneline <base_sha>..HEAD   # список коммитов в МР
```

Теперь у тебя полный контекст — и диф, и исходники в актуальном состоянии ветки МР.

## Шаг 5. Изучение дифа

Читай `.tmp/mr-<id>.diff` целиком, если он ≤ 2000 строк. Если больше — читай по секциям через offset/limit (см. правило про большие файлы).

Параллельно, по мере чтения, смотри исходники в worktree: чтобы понять контекст метода, найти вызывающих, проверить смежные файлы. Используй Grep/Read по конкретным именам.

Цели этого шага:
- Понять, что именно меняется
- Увидеть потенциальные проблемы самостоятельно (будешь сверяться с ревьюерами)
- Получить картину для **объяснения сути МР пользователю** (шаг 7)

## Шаг 6. Запуск ревьюеров

Запусти `/review` — он сам разошлёт quality / efficiency / architecture агентов параллельно.

**Всегда** запускай `/review`, независимо от размера МР. Даже если МР маленький — ревью обязательно.

Передай агентам:
- Путь к дифу: `.tmp/mr-<id>.diff`
- Путь к worktree для контекста
- Краткое описание сути МР (что меняется, ссылки на тикеты)
- **Явно попроси «только блокеры и важное»** — без nit'ов
- **Обязательно `mode=analysis-only`** — мы ревьюем чужой код, никаких правок и фикс-коммитов в ветку МР делать нельзя. orchestrator вернёт список FIX-рекомендаций, мы оформим их как черновики комментов в шаге 8.

Подробности про сам механизм — в скилле `/review`.

**Дождись результатов `/review` до перехода к шагу 7.** Шаг 7 включает блок «итог находок», который требует сводной информации от ревьюеров. Без их результатов шаг 7 выдастся с пустым итогом, и потом придётся дописывать отдельным сообщением.

## Шаг 7. Объяснение сути МР пользователю

**Самый важный шаг после сбора данных.**

Пользователь — программист, но **он не читал код этого МР**. Не делай вид, что он уже знает затронутые функции, файлы, переменные. Показывай релевантные куски кода, описывай что они делают. Это нужная для него информация — без неё он не сможет оценить замечания.

Что выдать в сообщении:

1. **Название МР + номера тикетов** (ссылки, если есть)
2. **Что делает МР в одном предложении** — на человеческом уровне, до технических деталей
3. **Зачем это делается** — мотивация из тикета (если получил содержимое на шаге 3)
4. **Как это делается технически** — последовательно, по частям:
    - Какие файлы затронуты и на каких правах (новые / изменённые / удалённые)
    - Ключевые изменения с примерами кода «было / стало» (короткие, по делу)
    - Побочные изменения (изменения visibility, новые классы, и т.п.)
5. **Что с тестами** — есть ли в МР, покрывают ли изменения
6. **Итог находок** — сколько блокеров, сколько важного, короткое содержание каждого (без деталей — детали в шаге 8)

### Правила объяснения

- **Не делать вид, что код уже в голове у пользователя.** Имена функций, имена классов, имена файлов — упоминай впервые с кратким контекстом: «в `Foo.bar()` — это метод который делает X».
- **Не использовать мелкие сокращения для МР-жаргона.** «`ImplA → ImplB` через `with(x)`» — расшифровывай: «вся логика теперь в интерфейсе, но через каст тянется к конкретной реализации».
- **Примеры кода — только когда нужны.** Предпочитай словесное описание. Если без кода не понять — давай 3-10 строк «было / стало», с коммитами и пояснением словами что именно поменялось.
- **Обычная программистская лексика — ок.** Класс, интерфейс, транзакция, smart-cast, DI — всё это пользователь знает. Не надо объяснять «что такое метод».
- **Низкоуровневые детали** (PRAGMA, fsync, WAL, JIT) — объясняй, ТОЛЬКО если они релевантны замечанию И пользователь не показал знакомство с темой. Если не уверен — кратко упомяни и будь готов развернуть по запросу.
- **Таблицы помогают.** Если сравниваешь «было / стало», «режим A / режим B», «плюсы / минусы» — таблица быстрее читается чем абзац.

После объяснения и списка находок задай пользователю 2-3 нумерованных вопроса в стиле `/remind-protocol` с вариантами и ⭐ Фаворитом. Типичные вопросы:

1. **Порядок обсуждения** — предложить свой (обычно «сначала простые блокеры, потом сложные», см. шаг 8) или пусть пользователь укажет
2. **Стиль комментов** — нейтральный / формальный / дружеский (фаворит обычно нейтральный)
3. **Глубина обсуждения каждого замечания** — сначала объяснение → затем черновик (фаворит), или сразу черновик

Ждать ответов до перехода к шагу 8.

## Шаг 8. Обсуждение находок по очереди

Порядок уже согласован в шаге 7. Если пользователь ответил «сам решай» — дефолт: простые/понятные блокеры (баги, очевидные ошибки) → архитектурные/структурные → важное.

Обсуждать **по одному замечанию за раз**. По каждому:

1. Объясни в чате **что за проблема** (детально, с кодом если нужно — это только для пользователя, не для коммента)
2. Предложи **черновик коммента** (короткий, см. шаг 9)
3. Жди одобрения / правок
4. Публикуй через `/txt`
5. Укажи **куда добавить** — `файл:строка`

Если у тебя появились сомнения в находке — **проверяй факты до того, как оформить коммент**. Запусти grep, прочитай соседние файлы, посмотри вызывающих. Лучше потратить лишние 30 секунд, чем опубликовать неверное замечание.

## Шаг 9. Стиль комментов

**Review-специфичное правило:** не навязывать решение. Укажи на проблему + спроси «продумывалось?» / «намеренно?» / «можно обсудить?». Автор сам решит как фиксить — у него больше контекста. Советы типа «обернуть в транзакцию», «вынести в класс» избегай — они звучат как указание. Исключение: если пользователь явно просит дать рекомендацию.

**Оформление и маскировка** — применяй `/remind-text-style` (нет backticks вокруг идентификаторов, нет длинного тире / умных кавычек, короче, один факт + один фикс).

**Способ выдачи коммента:**
- По умолчанию — выдай прямо в чат, пользователь скопирует из десктопа
- Если нужен отдельный файл (длинный коммент, много спецсимволов) — вызови `/txt`

После публикации пользователь скажет «ок, давай дальше» — переходишь к следующей находке.

## Шаг 10. Завершение

Когда все блокеры обсуждены:

1. Спроси, обсуждать ли важное или закончить
2. Если закончить — резюмируй итог ревью (коротко: сколько блокеров опубликовано, что пропущено)
3. Worktree `mr-<id>-review` по умолчанию **оставляй** (`ExitWorktree keep`) — пользователь может захотеть вернуться. Если явно просит удалить — `ExitWorktree remove`.

## Связанные скиллы

- **`/review`** — механизм параллельных ревьюеров (quality / efficiency / architecture). Всегда запускай из шага 6.
- **`/txt`** — оформление черновика коммента в файл для копирования. Используется на шаге 9.
- **`/remind-protocol`** — общие правила коммуникации (русский, нумерация, варианты с фаворитом). Соблюдается на протяжении всего флоу.
- **`/commit`** — не применимо напрямую (ничего не коммитим в чужой МР), но если нужно закоммитить локальные правки — через него.
