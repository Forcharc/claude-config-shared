# Подробный гайд по рабочему процессу

Это развёрнутая версия правил из CLAUDE.md. Краткие правила — там, подробности — здесь.

---

## Процесс работы с проблемами

### Сбор и формализация

1. **Собрать все замечания** в список, не теряя деталей
2. **Формализовать каждую проблему** — сделать чёткой и понятной
3. **Сохранить порядок** — перечислять в исходной последовательности
4. **Не упрощать и не обобщать** — сохранять все нюансы

### Приоритизация

- 🔴 **Критически важные и блокирующие** — фундаментальные решения
- 🟠 **Важные архитектурные** — значимые, влияющие на систему
- 🟡 **Средней важности** — зависят от базовых решений
- 🟢 **Требуют обсуждения** — нужны аргументы
- ⚪ **Низкий приоритет** — косметические

### Процесс решения задачи

1. Выбрать самую важную проблему
2. Рассмотреть 2-4 варианта решения
3. Обсудить каждый с примерами, за/против
4. Предложить фаворита
5. Получить решение
6. Зафиксировать в логе
7. Определить новые вопросы
8. Обновить список и приоритеты
9. Перейти к следующей проблеме

### Итерационный подход

```
Итерация 1:
├─ Собрать проблемы
├─ Приоритизировать
└─ Решить топ-1 проблему
    ├─ Варианты → Обсуждение → Решение
    └─ → Новые вопросы

Итерация 2:
├─ Обновить список (старые + новые)
├─ Переприоритизировать
└─ Решить топ-1
    └─ ...
```

## Документирование

### Когда создавать документы

- Накопилось много вопросов — создать лог решений
- Принято важное решение — зафиксировать
- Возникли новые вопросы — добавить в список
- Нужна структуризация — отдельный документ

### Структура лога решений

1. Исходные вопросы/проблемы
2. Что решили (с обоснованием)
3. Что не решили / частично решили
4. Новые вопросы
5. Объединённый список актуальных вопросов
6. Приоритизация

### Правила

- Не терять контекст — ссылки на код/документы
- Нумеровать вопросы
- Краткость + полнота
- Версионность — когда принято решение
- Проверка актуальности — после изменений проверять связанные документы

## Примеры хорошего флоу

### Обнаружена проблема в документе

```
❌ Плохо: сразу исправить без обсуждения

✅ Хорошо:
1. Собрать все проблемы
2. Формализовать и приоритизировать
3. Выбрать самую важную
4. Предложить варианты решения
5. Обсудить → получить одобрение
6. Внести изменения
7. Зафиксировать в логе
```

### Архитектурное решение

```
❌ Плохо: "Я думаю, лучше X"

✅ Хорошо:
1. "Рассмотрим 3 варианта: X, Y, Z"
2. Для каждого: описание, примеры, за/против
3. ⭐ Фаворит: Y, потому что...
4. 🏛 Архитектурно правильный: X (если отличается)
5. Ждать решения
```

## Принятие архитектурных решений — подробности

### Формат обсуждения

1. Рассмотреть несколько вариантов (минимум 2-3)
2. Для каждого: описание, примеры кода/структуры, за/против, связи с другими решениями
3. В конце: ⭐ Фаворит и 🏛 Архитектурно правильный
4. Не настаивать — если сомнения, обсудить

> **Почему фаворит в конце:** во время написания вариантов может появиться лучшее решение.
> **Зачем указывать "архитектурно правильный":** чтобы осознанно выбирать между "оптимально" и "правильно".

## Подробности оркестрации

### Фикс багов после ревью

`implementer` агенты **параллельно** (по одному на независимую группу фиксов). Каждому — автономное описание: баг, место (file:line), как чинить, границы правки.

**Исключения:**
- Один тривиальный фикс (~5-10 строк) → сама, без агента
- Зависимые фиксы → последовательно или одним пакетом одному агенту

### Исследовательский агент

Для изучения кодовой базы используй `research-writer` (кастомный). Он сам вычисляет путь к папке результатов и пишет отчёт в файл через Write tool. (Встроенные `Plan`/`general-purpose`/`Explore` запрещены — см. `~/.claude/CLAUDE.md`.)

### Передача данных между агентами

**Файлы вместо копипасты:**

1. `research-writer` сам пишет результат в `~/.claude/projects/<slug>/memory/agent-results/<topic>.md`
2. Следующему агенту — путь к файлу + инструкция прочитать

```
❌ Agent(prompt="Вот результат: <10KB текста>...")
❌ Agent(prompt="Прочитай .../tasks/abc123.output") — это транскрипт

✅ research-writer: "Исследуй X. topic=research-x"
   → возвращает: ~/.claude/projects/<slug>/memory/agent-results/research-x.md
✅ implementer: "Прочитай ~/.claude/projects/<slug>/memory/agent-results/research-x.md. Реализуй..."

Три звена (research → plan → impl):
✅ research-writer: topic=recall-arch       → recall-arch.md
✅ planner:         читает recall-arch.md   → recall-arch-plan.md
✅ implementer:     читает recall-arch-plan.md, реализует
```

`<slug>` — `pwd | sed 's|[/.]|-|g'` (путь проекта с заменой `/` и `.` на `-`). research-writer и planner вычисляют его сами.

**Куда писать:** `~/.claude/projects/<slug>/memory/agent-results/`. НЕ `/tmp`, НЕ `tasks/<id>.output`.

**Допустимо напрямую:** краткий результат (<500 символов), конкретный факт.

### Работа с неопределённостью

- Непонятно → рассмотреть варианты
- Автогенерация → явно указать, продумать
- Зависимость от другого решения → указать
- Оффтопные идеи → пометить "ИДЕЯ", включить в список

## Подробности: Tech Debt

### Что записывать

- Дублирование кода, копипаста
- Хардкод, магические числа/строки
- Отсутствие обработки ошибок
- Подозрительная хрупкая логика
- Устаревший код, deprecated API
- Недостающие тесты для критичной логики
- N+1 запросы, лишние аллокации, блокирующие вызовы
- Нарушение архитектурных принципов
- TODO/FIXME/HACK в коде
- Любые "а вот тут бы стоило..." мысли

**Порог — НИЗКИЙ.** Лучше записать лишнее. Даже подозрение — пометь 🟢.

## Подробности: Большие файлы

**Стратегия "Размер → Несколько строк → Паттерн → Якоря":**

1. `ls -lah file.log`
2. `head -30 file.log` / `tail -30 file.log`
3. `grep -n "ключевое" file.log | head -10`
4. Якоря на основе найденного паттерна

**Мониторинг фоновых команд:**
- ❌ BashOutput без фильтра — грузит весь лог
- ❌ Проверять каждые 5 секунд — жди минимум 1 минуту
- ✅ `tail -5 logfile.log` — строки могут быть длинными!
- ✅ Увеличивать постепенно (5 → 10 → 20)
